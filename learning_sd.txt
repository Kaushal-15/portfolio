Project Documentation Draft
1. Overview
This is a full-stack learning platform built around structured roadmaps, quizzes, adaptive exams, progress tracking, and a document-to-quiz workflow. The user-facing product is basically: pick a learning track, study through generated daily plans and roadmap lessons, take quizzes/tests, track XP/leaderboard progress, and optionally sit for managed exams.

The backend is doing most of the heavy lifting. The frontend is mostly a React UI layer over a fairly broad Express + MongoDB API.

2. Problem It Solves
The project is trying to combine two adjacent workflows in one product:

self-paced technical learning with roadmaps, quizzes, progress, and AI-generated study material
controlled exam delivery with admin-created exams, student verification, adaptive routing, and basic proctoring hooks
Why it exists, based on code:

to give learners a guided path instead of disconnected resources
to turn study material into quizzes automatically
to track progress/XP over time
to support both casual practice and more formal exam sessions
3. Tech Stack
Frontend: React 18, Vite, React Router, TailwindCSS, Framer Motion, Lucide icons, Three.js
Backend: Node.js, Express
Database: MongoDB via Mongoose
Exam storage: separate Mongo connection for exam collections via mongoose.createConnection
Auth: JWT in cookies and Authorization header support, refresh tokens, Passport Google OAuth, express-session
File upload: Multer
Document parsing: pdf-parse, mammoth
AI/LLM: Groq, Google Gemini, OpenAI SDK present
Email: Nodemailer for OTP verification
Rate limiting / security: Helmet, CORS, express-rate-limit
Scheduling: node-cron
Deployment-related: Vercel entrypoint (api/index.js), Dockerfiles for frontend/backend, nginx config in frontend

Honest read: the structure is workable for an MVP, but it is not very clean anymore. There are backup files, fix scripts, duplicate patterns, and multiple generations of logic sitting next to each other.

5. Core Features
Roadmap onboarding: users choose a track, skill level, and timeline, then the app stores roadmap preferences and uses them to drive the dashboard and learning screens.
Learning roadmap view: daily learning content is fetched and reshaped into week-based UI timelines, with roadmap lessons and project sections layered on top.
Quiz system: users can create quiz sessions by roadmap, difficulty, and topic. Question selection uses categories, question history, and some spaced-repetition style prioritization.
Adaptive quiz behavior: quiz sessions track per-question performance and adjust difficulty during the session.
Custom learning: users upload PDF/DOCX/TXT files, backend extracts text, chunks it, and generates quiz questions from that content.
Exam system: admins can create exams, upload questions, manage students, view results, and run static, dynamic, mixed, adaptive, and synchronized exam modes.
Biometric verification: students can upload a reference photo, admins approve it, and later a live photo is compared before exam entry.
Camera/proctoring hooks: there is camera monitoring and recording plumbing, plus fullscreen / tab-switch violation handling in the frontend.
AI chatbot: authenticated users can send short chat messages and get platform-aware responses.
XP + leaderboard: progress updates award XP, map users into leagues, track streaks, and expose leaderboard endpoints.
AI-generated learning content: content can be generated by topic/subtopic and cached.

7. Data / Schema Design
Main entities:

User: auth identity, role, onboarding fields, selected roadmap, refresh token, verification state
Roadmap: roadmap metadata, levels, lessons, quizzes, projects
Question: question bank with categories, difficulty, tags, validation metadata
Quiz: live or completed quiz session with embedded question state
UserProgress: per-user per-roadmap completion and XP
UserXP: global XP, league, streak, category XP
Document: uploaded source file metadata
Chunk: extracted document segments used for quiz generation
ContentCache: cached AI-generated learning content
Exam-side entities:

ExamMaster: exam definition and config
ExamQuestion: frozen exam questions
ExamSession: in-progress student session
ExamAttempt: submitted attempt summary
ExamLog: violations / session events
BiometricVerification, CameraRecording, AdaptiveDifficulty, ExamQuestionResult
Relationships:

User -> UserProgress is one-to-many by roadmap
User -> UserXP is one-to-one
User -> Quiz is one-to-many
User -> Document -> Chunk
ExamMaster -> ExamSession / ExamAttempt / ExamQuestion
Roadmap embeds lesson and project content rather than normalizing it heavily
No relational DB here. This is all Mongo documents, with some embedded arrays getting fairly large.

9. Core Workflow
Main learner flow:

user signs up or logs in
selects roadmap, level, and timeline
dashboard loads profile, progress, recent test results, roadmap history, topic analysis, and uploaded docs
learn screen fetches daily-learning content and roadmap metadata
user completes quizzes/tests
backend stores results, updates progress, and awards XP
Custom learning flow:

user uploads a file
backend stores file metadata in Document
backend extracts text and chunks it into Chunk
user requests quiz generation
backend concatenates chunk content and prompts Groq/Gemini, or falls back to content-based question generation
result is stored as a Quiz
Exam flow:

admin creates exam and either links existing questions, uploads raw questions, or configures dynamic generation
student joins with exam code
backend validates entry and creates/retrieves ExamSession
frontend renders exam session, timer, answers, and violation handling
backend logs events and submits final attempt
admin can review results, analytics, students, and verification status
10. AI / Agent Logic
What exists:

AI content generation for study modules
AI-assisted quiz generation from uploaded documents
AI chatbot
older OpenAI-based question generation and assessment services also exist in the codebase
What does not really exist:

no true agent system
no tool calling
no embeddings/vector DB
no retrieval pipeline beyond naive chunk storage and concatenation
no durable conversation memory beyond sending a short recent history array
How it is wired:

contentGenerator tries Groq first, then Gemini, then template fallback
quizGenerationService uses Groq or Gemini to generate MCQs from concatenated document chunks
chatbotController sends a fixed system prompt plus recent history
OpenAI-based AIClient and QuestionGenerationService appear more like an older service layer and test target than the main runtime path
Prompt/orchestration quality:

functional, but mostly prompt-in-controller/service style
response shaping is JSON-driven where possible
custom-learning generation is simple and pragmatic, not sophisticated RAG
chatbot context is shallow
some AI code is duplicated across services instead of centralized
11. Bottlenecks / Weak Spots
The backend has grown into a feature pile. examController.js and quizRoutes.js are doing too much and will be hard to maintain safely.
Auth is mixed. The app uses cookies, refresh tokens, sessions for Google OAuth, and also stores an access token in localStorage. That boundary is not clean.
Frontend auth architecture is partially migrated. AuthContext exists, checks /api/auth/me, and is not mounted in main.jsx, while most screens use /api/profile/me directly.
Route design is inconsistent. There are legacy and new progress routes side by side, and business logic leaks into routes in several places.
Proctoring is not production-grade. The frontend literally comments out the ProctoringService startup, hardcodes the violation limit, and forces requireCamera: true for all exams in ExamSession.
ProctoringService uses bind() in both add/remove listeners, so cleanup will not properly detach the original handlers.
Biometric verification is a placeholder. Similarity is based on base64 string length plus randomness. This is enough for a demo, not for real verification.
DELETE /api/content/cache/clear is public in current routing. That should not be public.
POST /api/xp/reset-weekly says admin only, but the controller explicitly notes admin restriction is still missing.
The content generator advertises template fallback, but contentRoutes returns 503 if no API key exists, so fallback is blocked at the route layer.
ContentCache only allows text|video|audio|image, but route validation accepts flashcards and full_module. That looks like a real schema mismatch.
There are many hardcoded assumptions: dashboard total lessons 45, exam violation threshold 3, forced camera behavior, default time-per-question, etc.
Logging/debug noise is high. There are many console.log traces, debug endpoints, and fix scripts still in the main repo.
The codebase has visible duplication. Example: UserProgress.submitProject is defined twice.
Transaction boundaries in XP logic are not especially strong. A session is started, but helper methods like getOrCreate are not consistently session-aware.
File storage is local uploads/, which is fine for local/dev but fragile on serverless or multi-instance deployments.
12. Improvements / V2
Split exam logic into smaller services: exam creation, session lifecycle, adaptive routing, analytics, and proctoring should not live in one controller.
Unify auth around one approach. Either go cookie-first or token-first, but not both.
Mount one real frontend auth provider and remove the dead path mismatch.
Move route-level business logic into service modules more consistently.
Replace biometric placeholder logic with an actual face-match provider or local model, or clearly downgrade it to “photo approval only.”
Lock down sensitive routes: cache clear, weekly XP reset, debug endpoints.
Replace local uploads/chunks with object storage if this is meant to run on Vercel or anywhere stateless.
Centralize AI provider selection and prompt construction. Right now Groq/Gemini usage is repeated across services.
If document-based quiz generation is important, add retrieval instead of concatenating chunks blindly.
Clean up duplicated/legacy code paths: old auth context, backup components, duplicate progress routes, duplicate model methods.
Add request validation more consistently. Some routes validate carefully, others accept broad request bodies.
Reduce hardcoded UI assumptions and read config from API where possible.
13. Recruiter Summary
Short project summary:
A full-stack learning platform that combines guided tech roadmaps, adaptive quizzes, document-to-quiz generation, XP tracking, and an admin-managed online exam system with basic proctoring and biometric verification flows.

Technical highlights:

Built a React + Express + MongoDB platform with both learner and admin workflows.
Implemented quiz/session models with adaptive difficulty and question-history-aware selection.
Added a custom-learning pipeline that uploads documents, extracts text, chunks content, and generates quizzes with Groq/Gemini.
Designed a separate exam subsystem with its own Mongo connection, session tracking, synchronized/adaptive modes, and event logging.
Added progress, XP, league, and leaderboard logic on top of roadmap-based learning flows.
What this project proves:
This project shows real full-stack product-building range: not just CRUD screens, but stateful session flows, AI integration, admin tooling, file ingestion, progress systems, and exam-style workflow design. It also shows the typical MVP tradeoff pattern clearly: strong feature ambition, working end-to-end flows, and a codebase that now needs refactoring and boundary cleanup before scale.

I analyzed this from the implementation and repo structure directly. I did not run the app or test suite in this pass, so the draft is based on static inspection rather than runtime verification.
